Version One

I am creating a book club database that shows books that members of the book club have read, information about the books they have read, as well as if the book has a screen adaptation and the rating of the book from members of the book club. My categories in my table include title, author, series, publisher, date published, isbn, number of pages, genre, rating and screen adaptation.


Version one is a trial and just uses a spreadsheet to look at how the data could be organised and classified.

    • Relevant Implications

    Cultural: IS the database culturally appropriate? What does this mean in the context of the database? Text/character sets in digital technology can be complicated. If a program does not recognise a character, it will replace it with a ? or a substitute A for Ā. A database needs to be tested to ensure it will properly store a wide range of character sets from different languages.

    Sustainability/Future Proofing: The database should be easy to maintain and easy to add new features to. Adding new features should not require major reconfiguration of the whole database The database should be designed to prevent duplicated of information and repetition of information. Because if the same information appears multiple times, any alterations to it will need to be done in multiple places in the database, and it is likely some inconsistencies could occur.

    End user considerations: Uses plain English communication to user (does not use “tech speak”) so tht the user can easily understand the information that is being presented to them.

    Reflection

  • In version 2 I split the database into 3 tables. I separated the author’s and the main data so that I could eliminate the problem of duplicate authors. The issue was that author’s such as Rick Riordan would come up twice, and I had to deal with the issue of multiple authors writing one book. In version 2, I have eliminated this problem by giving each author an author_id, so that they will now appear once. Through the association table and added author table I was also able to deal with the problem of having multiple authors that wrote one book.

  • There is also still the issue with series as there is a lot of empty fields, and this is not ideal.


  • There was an issue with the datasheets which caused a few issues with the database. These were issues such as a space after a word. So I cleaned up the sheet by removing any tailing spaces.


  • There was an issue in the database where the isbn number came up with a .0 on the end of it when the csv was imported to the database. The issue was happening because SQL was automatically setting ISBN as a real text. I figured out that I could fix this issue however by disabling data type detection and as I set the dataypes in my table myself it was all fine.


  • There was also an issue however where when you search the first name of some authors nothing turns up but when you search the last name it works. It only happened for author_id's 7,8,11,13 and 17 and there seemed to be no pattern. After some testing however, I figured out the problem and fixed it. To figure out the problem I went through and tested the "like and %" query style and found that the author appeared when this query was searched, so I figured that the issue was that there was a space after the Authors' name in the original datasheet. I went got rid of all spaces after first and last names and spaces and it now works.


  • In version 2 of my database it only shows 2 books that have the genre YA Fiction. However, in our actual database, there are more than 2 books that have the genre YA Fiction, but as they also have other genres, the database can't detect the genre YA Fiction. Also, when you look up just "YA" or "fiction" nothing appears. In the next version I will create a seperate table to manage the multiple genres so I can assign multiple genres to one book. Right now, when you search up more than one genre it does not show up. It is also because the genre's are split with commas which the program doesn't separate.


  • I also tested code to insert a new author and a new title into the database as well as deleting an author and a title. I struggled with this at first because I first tried to do all values at once. However, this was impractical and confusing, so I seperated and organised all of the information I wanted to put in into sepereate sections.

  • When making version 3 with the additional genre table, I tested out two versions. The additional genre table means that multiple genres can be assigned to one title. This means that the title will now appear when the user searches for more than one genre. For example, in my queries "Red Queen" appears when the genre fiction is searched as well as fantasy. Assigning multiple genres makes it easier to classify books, as well as in the context of a book club, the user would not have issues finding all books under a certain genre where the book has more than one genre.


  • Version one of the genre table assigned a genre to each book then listed yes or no for YA genre and fiction genre. I did this because YA and fiction were genres that nearly every book included, and then each book generally only had one other genre. However, this design was not easy to change if adjustments were to be made and had some issues with repeating values.


  • I ended up going with my second version of the genre table. The second version of the genre table had a list with all the genres and assigned them all with unique ids. I then went an assigned the relevant genre ids to the title ids in a seperate association table. This design means that it is a lot easier to change and enter data. With the genre table now, I can now make a query (as seen before) where I can search up the genres YA and fiction together, and the result will show up.


  • In the future it would be good if I made another table to keep track of book owners. As the context for my database is a database of books for a book club, it could be useful for the user to know who is reading what book/who currently owns what book if the books are shared across all book club members.


  • It could also be interesting to add in information about Author's in the author table or other books that the author has written. Often if people like a book by a certain author they like to read other books under that author too.


  • Another possible addition is a table showing the names of the screen adaptations of the books and the rating that they got. A lot of people like to watch the tv or movie version of books to help them visualise the characters or storyline better, so adding this into the database would be useful for the Book Club members.

    • Relevant Implications Summary

  • Cultural With so much of our society today including a wide range of diverse cultures, it is important that we embrace all aspects of cultures, respect all languages and names. With the context of my database being a book club, it may include book club members from a wide range of cultures, who may want to share books from their culture with the rest of the book club. To respect all cultures and people's names, the database should accept all letters including those with accents on them. This means that all cultures will be treated fairly and their names will be respected and not culturally appropriated. In my database, I have addressed this issue by inserting an author with an accent in their name and testing to see if the database recognises it and shows the author when searched. In the query above you can see that I tested this using Émile Gaboriau who is a French author. The query has an É in the input and the output also has an É showing that the database includes accents and does not replace it with a substitue or come up with an error message. I also inserted a book by a Japanese author into the database as well and tested it to see what happened (as can be seen above). As shown in the table, even though the Japanese characters are not in english they still show up in the output when searched, which means it is likely to work with most other languages too. This shows that my database allows languages from all cultures.

  • Sustainability/Future Proofing It is important to consider how easy it is to add or change information when designing the database, so there will be less issues in the future. In my database, I have future proofed by drawing out diagrams, making plans and by designing tables that will be able to be changed easily. In all of my versions I made sure to write out a brief description about what is being added in the new version as well as making a plan for all versions. For versions 2 and 3 I drew a diagram (as can be seen above) showing the different tables and the columns that will be included in each of the tables. This is relevant to Future proofing because it means if there is an issue with the table I can refer back to my diagram and it will help me to identify the issue easier if changes need to be made. As well as this, it is easy to add new information into my database. The user just has to use "insert" and fill out the values. This is a lot simpler than having to reupload a new csv. It is also easy to assign author's with multiple books/multiple authors to one book and multiple genres to one book through my usage of association tables. The user will just have to select the relevant genre_id/author_id and match it to the title_id. This is a lot more efficient than using a design where you have to type in each genre/author in order to connect it to the main title table every time a new book is added. The design of the database also helps if the user wishes to make further improvements. As seen in version 2 and 3, it is easy to create a new table and connect it to the rest of the database using an association table. And because the data tables are specified into distinct "entities" (author, title, genre) it could be easy further tables (like readers).

  • End User considerations In my database, I have addressed end user considerations by using simple easy-to-understand language throughout my database. All of the column names are common words in the english language that people are likely to be familiar with, and most of the words used are plain and simple not "tech speak". I think the database as a whole would be quite easy to navigate for someone who has not much experience with databases. As well as this, the database suits the needs of a book club well. It contains the key information that would be useful for a book club and it's design makes it easy for the user to search up books.
